# Artist Profile Card — Repository Review
## Section 9: Reuse / Change / Replace Assessment

**Status:** Discovery only — no code changes made or authorized by this document.
**Purpose:** Assess current implementation against the approved target architecture (`ARTIST_PROFILE_CARD_ARCHITECTURE.md` §8: `SCAN → ARTIST PROFILE CARD → INTELLIGENCE PROCESSING → ATHENA™ → MISSION CONTROL`).
**Method:** Direct code reads only, same evidentiary standard as PR #359/#360 (file:line citations, no unverified claims). Where a claim in the architecture doc could not be independently confirmed, that is stated explicitly below rather than assumed true.

---

### 1. CIO/CIM Pipeline — **CHANGE**

`normalizeAuditResponse()` (`api/lib/normalizeAuditResponse.js:21`) builds the flat `canonical` object from raw engine output — pure field-normalization logic, no dependency on the CIO/CIM shape, directly reusable underneath an Artist Profile Card. `assembleCio()` (`api/_lib/cio-assembler.js:118`) builds the intermediate `cio` (schema: `api/schema/cio.js`), which `runRIE()` (`lib/rie/index.js:308`) feeds through `runIntelligenceEngine(cio, ALL_RULES)` and 9+ domain assemblers into `assembleCIM()` (`lib/rie/index.js:100`) — a 13-object model per `CIM_OBJECTS` (`api/schema/canonical-intelligence-model.js:20-32`). `certifyCIM()` (`lib/rie/certify.js:33`) deep-freezes the result and throws if any §8.2 key is missing. The output shape does not conflict with an Artist Profile Card — CIM is already "one artist, one frozen intelligence object" — but the pipeline currently *terminates* at `canonical`/`cim`, flattened back together by `CimAdapter.buildCimEnrichment()` (`lib/rie/CimAdapter.js:61,115-134`) for Runtime Context, not any Artist Profile Card layer. Becoming an Artist Profile Card feeder requires a new consolidation step after `certifyCIM()` that also merges onboarding data (`profiles.music_rights_profile`) — today merged only downstream, client-side, at Runtime Context (`mission-control.js:1361-1379`, per architecture doc §4.2) — for which CIO/CIM has no pipeline today (confirmed: no write-back path from onboarding into `cio-assembler.js`/`rie/index.js` found).
**Biggest risk:** `CimAdapter.js` is a flattening/backward-compat shim, not a clean handoff boundary — inserting an Artist Profile Card assembly step correctly (after CIM, before flattening) versus incorrectly (grafted onto the already-flattened `canonical`) is an easy place to accidentally create a second, divergent canonical shape.

### 2. Runtime Context (`buildWorkspaceRuntimeContext()`) — **CHANGE**

Per PR #360's Field-Level Trace Report, the function (`public/js/runtime-context-mapper.js:102-159`) takes a flat `canonical`-shaped `payload`, a separately-sourced `musicRightsProfile`, and a caller-computed `derivedState`, returning a new 23-field object written to `sessionStorage['royalte_workspace_context']` (`mission-control.js:1481`) with exactly one confirmed production caller (`mission-control.js:1472-1480`). Per architecture doc §3 (Board decision 2026-07-18), Runtime Context is **not replaced** by the Artist Profile Card — it becomes a downstream reader of it. Its first two arguments change source (Artist Profile Card object instead of raw `canonical` + independently-fetched `musicRightsProfile`), but its output contract (the Board-locked 23-field `royalte_workspace_context` shape, Runtime Context Contract v1.2/1.3) need not change, since every workspace reads only that output shape, never `canonical` directly (zero workspace files reference `canonical`/`cim`, PR #360 §18). The trace report also documents that this function already silently drops `canonical.health`, `canonical.cim` (beyond `.identity`), and `canonical.source` (§14) — worth resolving during cutover rather than carrying forward.
**Biggest risk:** Because most output fields are same-object references, not clones (§16 of the trace report), swapping the input source without re-verifying every `_r()` fallback path risks silently reintroducing one of the three already-known drops, or a new one, without any workspace visibly breaking (workspaces null-guard silently).

### 3. Mission Control Workspaces (`public/workspaces/*.html`) — **REUSE**

There are **12** files under `public/workspaces/` (not ~10-11 as scoped — `ls` confirms: `ai-insights.html`, `backend-intelligence.html`, `catalog-intelligence.html`, `executive-brief.html`, `global-music-footprint.html`, `health-intelligence.html`, `identity-intelligence.html`, `media-intelligence.html`, `monitoring-timeline.html`, `priority-actions.html`, `publishing-intelligence.html`, `royalte-review.html`; `media-intelligence.html` postdates PR #360's 11-file count). All wired workspaces read exclusively via `window.RoyalteContext.readWorkspaceContext({contract: '<name>'})` (`public/js/mc-workspace-context.js:277-312`), which parses `sessionStorage['royalte_workspace_context']` — none call `fetch()` or Supabase directly (confirmed by PR #359 §6). Explicit lock language was found inline in `identity-intelligence.html:519` ("ARCHITECTURE (Board-locked 2026-07-12)") and `media-intelligence.html:729` ("per the Board-locked Runtime Context Contract"); other locked workspaces (`publishing-intelligence`, `catalog-intelligence`, `backend-intelligence`, `global-music-footprint`, `ai-insights`) carry their lock status in `governance/`/memory rather than inline file comments, per this session's memory index. Because every workspace's data contract is already the Runtime Context shape rather than `canonical`/CIM directly, and Runtime Context itself is not being replaced (see §2), **workspace HTML/JS requires no change at all** for the Artist Profile Card migration as currently scoped — the insulation the Board already built (Runtime Context as the sole delivery contract) is exactly what makes this a Reuse, not a Change.
**Biggest risk:** This Reuse classification holds only as long as Runtime Context's *output* shape stays frozen; if the Artist Profile Card migration later changes `royalte_workspace_context`'s 23-field contract (not authorized today), all 12 files become in scope simultaneously, including several individually Board-locked layouts.

### 4. ATHENA (`api/athena/`) — **UNBUILT (target layer, not yet wired)**

`createAthenaEngine({ missionControlApi })` (`api/athena/index.js:21`) has **zero production callers**, confirmed independently in this pass: `api/mission-control-integration/graceful.js` and `api/executive-brief/recommendations.js` do call it, but both live inside directories with zero production importers themselves (`governance/PLATFORM_REFACTORING_DISCOVERY_REPORT.md:40,68`; `governance/PLATFORM_DEAD_CODE_REPORT.md:16`: "`createAthenaEngine()` has zero importers"). **Note:** the architecture doc (§5) cites `ERI_PHASE1_ARCHITECTURE_ASSESSMENT.md` in `governance/` as a second confirming source — this file **could not be located**; no file by that name exists there. The zero-caller claim is nonetheless independently confirmed via the two files above. What exists: a fully-built 12-file engine (`analysis.js`, `risk-analysis.js`, `opportunities.js`, `recommendations.js`, `confidence.js`, `insights.js`, `validate.js`, `prompts.js`) designed against Mission Control Data API envelopes, not CIO/CIM directly (`api/athena/index.js:2-8`: "Consumes only through the Mission Control Data API™... must never import from api/monitoring, api/resolution, api/normalization, api/orchestrator, api/registry, or api/evidence directly"). A real, substantial Reuse candidate for the "ATHENA" stage — but gated on the Mission Control Data API™ (`api/mission-control-api/`), itself one of the confirmed-dead, zero-production-caller directories.
**Biggest risk:** ATHENA's own constitutional constraint (input only via Mission Control Data API envelopes) may itself need to change or be reinterpreted once Artist Profile Card becomes its intended input — that's a second migration decision, not a formality, since the engine's internal contract was designed around a different upstream shape than the one now proposed to feed it.

### 5. Current AI/Intelligence Pipeline (`royalte-ai-assembler.js`, `lib/rie/`) — **REUSE, then supersede at the ATHENA boundary**

`api/_lib/royalte-ai-assembler.js` is explicitly deterministic, template-based, no-LLM ("`generatedBy: 'engine_template'`", header lines 20-27) and documents its own position in the constitutional chain: `Scan Engine → Domain assemblers → Royaltē AI™ Assembler → audit_scans.payload.royalteAI → Mission Control` (lines 5-16). It is genuinely live and load-bearing — the `royalteAI` domain assembler called from `runRIE()`'s pipeline (PR #359 §1 step 9, §5), read by `ai-insights.html` (PR #360 §18/§19). Architecture doc §5 is explicit that ATHENA "supersedes the current `royalteAI` assembler pipeline... once the migration reaches that stage" — a scheduled decision, not an immediate one. `lib/rie/` broadly (RIE orchestration, domain assemblers, `certify.js`) is the "Intelligence Processing" stage's closest existing analogue and a strong Reuse candidate for that stage specifically, distinct from the AI-narrative assembler which is what ATHENA specifically targets.
**Biggest risk:** Conflating "Intelligence Processing" (the target stage, closest analogue = `lib/rie/`'s domain-assembler orchestration) with "ATHENA" (the target stage, closest analogue = `api/athena/`) risks retiring `royalte-ai-assembler.js` prematurely, before `api/athena/` has a real upstream input — the architecture doc's non-goals (§6) explicitly forbid renaming/retiring the existing engine in this phase; that constraint is easy to violate mid-migration if the two target stages get merged in implementation.
